# Phase 07.2: Per-Channel Thompson Sampling + Office-Scoped Candidates — Research

**Researched:** 2026-04-15
**Domain:** PostgreSQL schema migration, query rewrite, multi-tenant RLS, Thompson sampling
**Confidence:** HIGH — all findings verified against current code in this repo

## Summary

Phase 07.2 is a targeted database + service-layer fix for two bugs shipped by 07.1:
G-01 (`get_candidate_pool` has no office scope) and G-02 (`restaurant_stats` is keyed
by `(restaurant_id, workspace_id)`, not by channel). Both are enforced by existing
Postgres constraints (unique index + RLS), so migration 009 must simultaneously add
`restaurants.location_id`, re-key `restaurant_stats`, and update RLS. The hardest part
is not the SQL — it is **threading `channel_id` through the call chain**
(`push_poll → ensure_poll_options → update_stats_lazy → get_candidate_pool /
update_restaurant_stats`), none of which currently carries it.

**Primary recommendation:** Single Alembic revision 009 chained from 008. Drop and
recreate `restaurant_stats` rather than ALTERing (D-04 wipes it anyway, and recreating
gives a clean PK + FK + RLS pattern). Thread `channel_id` end-to-end through
`push_poll → ensure_poll_options → update_stats_lazy` as an explicit parameter,
defaulting to `None` only for legacy call sites (scheduled jobs will need updating).

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01** — `restaurants` gains a `location_id` FK column referencing `workspace_locations.id`. Candidate pools are office-scoped via this column.
- **D-02** — `restaurant_stats` is re-keyed by `(channel_id, restaurant_id)`. Thompson sampling is per-channel, not per-office and not per-workspace.
- **D-03** — Migration 009 executes in three steps: (a) add `restaurants.location_id` nullable, (b) backfill existing rows to each workspace's default office (`workspace_locations.is_default = true`), (c) set `restaurants.location_id` NOT NULL with FK to `workspace_locations.id`.
- **D-04** — On migration 009, wipe existing `restaurant_stats` and start every `(channel_id, restaurant_id)` pair at Beta(1, 1). No cloning, no even-split.
- **D-05** — FK `restaurants.location_id → workspace_locations.id` is declared `ON DELETE CASCADE`.
- **D-06** — No retroactive replay of `polls` history. Phase 4's reputation job emits per-channel rows going forward only.
- **D-07** — Both seed paths (`setup.py` install, `slack_actions.py` add-office modal) MUST tag newly seeded restaurants with the owning `location_id`.
- **D-08** — `get_candidate_pool(poll_date, channel_id)` resolves `channel_id → location_id` via `channel_locations`, filters `r.location_id = :location_id`, joins `restaurant_stats` on `(channel_id, restaurant_id)`, falls back to Beta(1, 1) when missing.
- **D-09** — Phase 4 reputation/stats-processing job must emit `(channel_id, restaurant_id)` rows.
- **D-10** — Regression tests MUST cover G-01 isolation (two channels, two offices) and G-02 divergence (two channels, same office, independent posteriors).

## Migration 009 Shape

Style conventions observed in 007/008:
- Raw `op.execute(...)` SQL (no `op.add_column`/`op.create_table` helpers).
- RLS pattern: `ENABLE`, `FORCE`, `CREATE POLICY tenant_isolation ... USING (... = current_setting('app.current_tenant', true)) WITH CHECK (...)`.
- Grant DML + sequence USAGE,SELECT to `lunchbot_app` inside a `DO $$ ... IF EXISTS pg_roles $$`.
- Inline comments referencing the decision IDs (T-07.1-NN style).
- Backfills guarded with `NOT EXISTS` for idempotency.

### Step A — `restaurants.location_id` (G-01)

```sql
-- A1: add nullable
ALTER TABLE restaurants ADD COLUMN location_id INTEGER;

-- A2: backfill using each workspace's is_default office (D-03b).
-- 07.1 migration 008 guarantees exactly one is_default office per tenant that
-- had a legacy workspaces.location; for tenants that had none, no restaurants
-- exist yet either, so LEFT JOIN is safe.
UPDATE restaurants r
SET location_id = wl.id
FROM workspace_locations wl
WHERE wl.team_id = r.workspace_id
  AND wl.is_default = TRUE;

-- A3: any leftover rows (workspace_id without a default office) are orphans
-- from partial installs. Refuse to proceed rather than silently drop data.
-- Migration aborts with a clear error via a single SELECT:
DO $$
DECLARE orphan_count INTEGER;
BEGIN
    SELECT COUNT(*) INTO orphan_count FROM restaurants WHERE location_id IS NULL;
    IF orphan_count > 0 THEN
        RAISE EXCEPTION '% restaurants have no default office; fix before running 009', orphan_count;
    END IF;
END $$;

-- A4: lock it down
ALTER TABLE restaurants
  ALTER COLUMN location_id SET NOT NULL,
  ADD CONSTRAINT restaurants_location_id_fkey
    FOREIGN KEY (location_id) REFERENCES workspace_locations(id) ON DELETE CASCADE;

-- A5: index for the new filter (candidate pool query hits this every poll)
CREATE INDEX idx_restaurants_location_id ON restaurants(location_id);
```

### Step B — `restaurant_stats` re-key (G-02)

Because D-04 wipes the table, the cleanest path is **DROP + CREATE**, not ALTER.
It avoids mutating RLS policies in place and eliminates the legacy
`workspace_id` column entirely.

```sql
-- B1: drop the entire table (cascades RLS, grants, seq, policy, UNIQUE, FKs).
DROP TABLE restaurant_stats CASCADE;

-- B2: recreate with per-channel key. Note channel_id is VARCHAR(64) to match
-- channel_locations.channel_id; we do NOT add a FK to channel_locations
-- because rows should outlive a channel unbinding/re-binding within the same
-- office (Thompson state is a property of the channel, not of the binding).
CREATE TABLE restaurant_stats (
    channel_id    VARCHAR(64) NOT NULL,
    restaurant_id INTEGER NOT NULL REFERENCES restaurants(id) ON DELETE CASCADE,
    team_id       VARCHAR(64) NOT NULL,  -- needed for RLS; see note below
    alpha         FLOAT DEFAULT 1.0,
    beta          FLOAT DEFAULT 1.0,
    times_shown   INTEGER DEFAULT 0,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (channel_id, restaurant_id)
);

CREATE INDEX idx_restaurant_stats_restaurant_id ON restaurant_stats(restaurant_id);

ALTER TABLE restaurant_stats ENABLE ROW LEVEL SECURITY;
ALTER TABLE restaurant_stats FORCE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON restaurant_stats
FOR ALL
USING (team_id = current_setting('app.current_tenant', true))
WITH CHECK (team_id = current_setting('app.current_tenant', true));

-- B3: grants identical to 003
DO $$
BEGIN
    IF EXISTS (SELECT FROM pg_roles WHERE rolname = 'lunchbot_app') THEN
        GRANT SELECT, INSERT, UPDATE, DELETE ON restaurant_stats TO lunchbot_app;
    END IF;
END $$;
```

**RLS tenant column decision:** `channel_id` alone is not sufficient for RLS — the
tenant GUC is a `team_id`. Options:

1. **Keep a denormalized `team_id` column** (shown above). Matches the
   `channel_locations` pattern in migration 007 exactly. Simple. Chosen.
2. Resolve via join to `channel_locations` in the policy. Rejected: policies that
   subquery are fragile and slow.

The planner should confirm option 1 but it is the path of least resistance and
mirrors the pattern 07.1 already introduced.

### Migration 009 ordering constraints

1. Step A (restaurants.location_id) MUST run first. Step B's `DROP TABLE ... CASCADE`
   does NOT touch `restaurants`, but keeping them ordered makes the failure-mode
   story simple: if A aborts on orphans, stats table is still intact.
2. The D-04 wipe is **implicit in Step B's DROP** — no separate `DELETE FROM` needed.
3. Downgrade path: recreate `restaurant_stats` with the old shape (workspace_id PK)
   and drop `restaurants.location_id`. Downgrade is an emergency-only op — best-effort
   is acceptable per the 008 precedent.

### ON CONFLICT semantics gotcha

`save_restaurant` today does `ON CONFLICT (place_id) DO UPDATE`. If the same
place_id is seeded under two different offices (possible in a workspace with
overlapping geographic zones), the UPSERT will overwrite `location_id`.
**CONTEXT deferred section says "Cross-office restaurant sharing (explicitly
not supported — each restaurant belongs to exactly one office)."** So the
current `ON CONFLICT (place_id)` unique constraint is fine — the same physical
restaurant can only live in one office. The planner should:

1. Decide: on conflict, keep existing `location_id` (first office to seed it wins)
   OR overwrite with the new one (last-seeded wins). Recommendation: **keep
   existing** via `ON CONFLICT (place_id) DO UPDATE SET ... location_id = COALESCE(restaurants.location_id, EXCLUDED.location_id)`. Rationale: if the first office
   already owns it, the second office has nothing to gain from stealing it, and
   stats + polls already point at the first `(channel, restaurant_id)` pairs.

## Reputation Job Location

Phase 4's scheduled reputation job is **`lunchbot.services.recommendation_service.update_stats_lazy`**,
called by `ensure_poll_options` (recommendation_service.py:159), which itself is
called by `lunchbot.services.poll_service.push_poll` (poll_service.py:220) — which
is triggered inline on every poll build (manual `/lunch`, scheduled cron,
add-office modal, etc.). There is **no separate scheduled job** for reputation
aggregation — it is lazy, triggered by the next poll that happens to need
candidates. This is important: the rewrite does not need to touch APScheduler
jobs, only `update_stats_lazy` and its callers.

**Write resolution strategy for `update_stats_lazy`:** read `channel_id` from
`polls.slack_channel_id` for each unprocessed poll. The stats-write loop becomes:

```python
for poll in unprocessed:
    channel_id = poll['slack_channel_id']
    if not channel_id:
        logger.warning('poll %s has no channel_id, skipping stats update', poll['id'])
        db_client.mark_poll_stats_processed(poll['id'])
        continue
    vote_shares = db_client.get_poll_vote_shares(poll['id'])
    ...
    db_client.update_restaurant_stats(
        channel_id=channel_id,
        restaurant_id=vs['restaurant_id'],
        alpha_increment=alpha_inc, beta_increment=beta_inc,
        team_id=workspace_id,
    )
```

This requires `get_unprocessed_polls` to return `slack_channel_id` in its SELECT
(currently returns only `id, poll_date, workspace_id` — db_client.py:302).

## New `get_candidate_pool(poll_date, channel_id)` shape

```sql
WITH channel_office AS (
    SELECT location_id
    FROM channel_locations
    WHERE team_id = current_setting('app.current_tenant', true)
      AND channel_id = %(channel_id)s
)
SELECT r.id AS restaurant_id, r.name, r.place_id, r.types, r.geometry,
       COALESCE(rs.alpha, 1.0) AS alpha,
       COALESCE(rs.beta, 1.0)  AS beta,
       COALESCE(rs.times_shown, 0) AS times_shown
FROM restaurants r
LEFT JOIN restaurant_stats rs
       ON rs.restaurant_id = r.id
      AND rs.channel_id    = %(channel_id)s
WHERE r.rating IS NOT NULL
  AND r.location_id = (SELECT location_id FROM channel_office)
  AND r.id NOT IN (
      SELECT po.restaurant_id
      FROM poll_options po
      JOIN polls p ON p.id = po.poll_id
      WHERE p.poll_date = %(poll_date)s
  )
ORDER BY r.name
```

Notes:
- The `COALESCE` fallback satisfies D-08 Beta(1,1) priors when the
  `(channel_id, restaurant_id)` row is missing.
- If `channel_office` returns zero rows (unbound channel) the outer query
  returns empty. Callers already handle empty candidate pools at
  recommendation_service.py:173–175.
- Still uses `execute_with_tenant` — RLS on both `restaurants` (workspace_id)
  and `restaurant_stats` (team_id) continues to enforce tenant scope.

## Risks & Pitfalls

1. **`polls.slack_channel_id` may be NULL for scheduled polls.** If the column
   is nullable and scheduled poll insertion does not set it, `update_stats_lazy`
   cannot determine which channel the stats belong to. **Planner must verify
   `slack_channel_id` is always set** before implementing. If it isn't, either
   (a) make it NOT NULL going forward via a small schema change in 009, or (b)
   warn + skip nullable rows (data loss acceptable per D-04 wipe philosophy).
2. **Channel rebinding orphans.** If a channel is unbound from office A, rebound
   to office B, its `restaurant_stats` rows still reference office-A restaurants
   which are no longer in its pool. Harmless (the candidate-pool query filters by
   `r.location_id`), but the rows become dead weight. Acceptable for v1; note
   for a follow-up cleanup job.
3. **Office delete cascade chain.** `restaurants.location_id ON DELETE CASCADE
   → workspace_locations`. `restaurant_stats.restaurant_id ON DELETE CASCADE →
   restaurants`. So deleting an office cascades to restaurants AND their stats.
   This is the intended D-05 behavior — verify in test case 4 above.
4. **RLS tenant column must exist.** The recreated `restaurant_stats` table
   needs a `team_id` column for RLS (see Migration Step B). Writers must set it.
   If a writer forgets, the INSERT fails the RLS `WITH CHECK`. The planner
   should make this a mandatory param to `update_restaurant_stats`, falling
   back to `g.workspace_id` only as a convenience.
5. **SQLite vs Postgres.** This project uses **Postgres only** (config `test`
   DATABASE_URL is a Postgres URL; migrations use Postgres-specific `RLS`,
   `JSONB`, `DO $$ ... $$`, `current_setting`, `ON CONFLICT ... WHERE`). No
   SQLite concerns.
6. **Concurrent-write hazards during migration.** Migration 009 runs inside
   Alembic's transaction. `DROP TABLE ... CASCADE` takes `ACCESS EXCLUSIVE`;
   other writers will block. The deploy script takes the app down first
   (blue-green) so this is fine — but do not run 009 against a live app.
7. **Alembic path confusion.** BRIEF/CONTEXT mention `lunchbot/alembic/versions/`
   but the actual directory is `migrations/versions/` at repo root. Plan 009
   must be placed at `migrations/versions/009_<slug>.py`.
8. **`emoji_service.update_database` bulk emoji tagging** (emoji_service.py:36–44)
   calls `db_client.add_emoji(place_ids, emoji_string)` which updates rows by
   `place_id` across the whole table. Post-migration, this is still safe
   because emoji tagging is per-place_id and not per-office. But a note: if
   two offices have the same place_id (which we've said is not supported),
   this bulk update would paint both. Matches current behavior, not a regression.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tenant isolation on the new `restaurant_stats` shape | Manual `WHERE team_id = ?` in every query | Postgres RLS policy mirroring migration 007 | Already the project convention; RLS enforces at-row-level even if a query forgets. |
| Channel-location resolution in the new query | Hand-join in Python | Inline CTE against `channel_locations` | One round-trip, respects RLS. |
| Beta(1,1) fallback for missing stats rows | Python-side defaulting | SQL `COALESCE(rs.alpha, 1.0)` in the SELECT | Already the 04-RESEARCH pattern; preserves LEFT JOIN semantics. |
| Mutating RLS policy in place during migration | `ALTER POLICY` | `DROP TABLE CASCADE` + `CREATE TABLE` (D-04 wipes anyway) | Simpler, no policy-rewrite edge cases. |
