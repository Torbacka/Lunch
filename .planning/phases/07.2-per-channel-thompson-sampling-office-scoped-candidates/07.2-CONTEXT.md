# Phase 07.2: Per-Channel Thompson Sampling + Office-Scoped Candidates — Context

**Gathered:** 2026-04-15
**Status:** Ready for planning
**Source:** 07.2-BRIEF.md + inline decisions captured during /gsd-plan-phase (expanded 2026-04-15 after research surfaced scheduler coupling)

<domain>
## Phase Boundary

Fix two live bugs surfaced by 07.1 UAT (G-01, G-02):

- **G-01** — `get_candidate_pool` filters restaurants only by `workspace_id`, so a channel bound to office A can be polled with restaurants from office B. `restaurants` has no `location_id` column.
- **G-02** — `restaurant_stats` is keyed at the workspace (or workspace+restaurant) level, so two channels bound to the same office share and contaminate each other's Thompson sampling priors.

In scope: schema migration 009, seed-path updates (including add-office modal seeding), `get_candidate_pool` rewrite, Phase 4 reputation job update to emit per-channel rows, regression tests for isolation and divergence, **AND** full channel-scheduling restructure: new `channel_schedules` table, scheduler_service refactor to channel-keyed jobs, App Home schedule UI becomes per-channel, `polls.slack_channel_id` NOT NULL, `upsert_suggestion` signature change, data-migration of existing workspace-level schedules.

Out of scope: reputation algorithm changes, retroactive reprocessing of historical polls, cross-office restaurant sharing.

</domain>

<decisions>
## Implementation Decisions

### Data model (locked from BRIEF)

- **D-01** — `restaurants` gains a `location_id` FK column referencing `workspace_locations.id`. Candidate pools are office-scoped via this column.
- **D-02** — `restaurant_stats` is re-keyed by `(channel_id, restaurant_id)`. Thompson sampling is per-channel, not per-office and not per-workspace. Rationale: each Slack channel is a distinct team; their votes must not pollute another team's posteriors even when they share an office.
- **D-03** — Migration 009 executes in three steps: (a) add `restaurants.location_id` nullable, (b) backfill existing rows to each workspace's default office (`workspace_locations.is_default = true`), (c) set `restaurants.location_id` NOT NULL with FK to `workspace_locations.id`.

### Stats backfill strategy (D-04) — answered inline

- **D-04** — On migration 009, **wipe existing `restaurant_stats` and start every `(channel_id, restaurant_id)` pair at the uninformative prior Beta(1, 1)**. No cloning, no even-split. Rationale: simplest path, no history bias, and the live system has low enough traffic that uninformative priors will re-converge quickly. Losing historical signal is acceptable; contaminating new channels with the wrong history would not be.

### Office-delete cascade (D-05) — answered inline

- **D-05** — FK `restaurants.location_id → workspace_locations.id` is declared `ON DELETE CASCADE`. Deleting an office deletes its restaurants, which in turn cascades to `restaurant_stats` rows via their own FK on `restaurant_id`. Matches the 07.1 delete-office UX (already lets admins remove offices) and avoids orphan rows. Success criterion 3's NOT NULL requirement is preserved because no location-less restaurants can exist.

### Retroactive replay (D-06) — answered inline

- **D-06** — **No retroactive replay of `polls` history.** Migration 009 only does the schema change + wipe from D-04. Phase 4's reputation/stats-processing job starts emitting per-channel `restaurant_stats` rows going forward, on each poll finalize. Historical polls do not get re-aggregated. Rationale: scope stays contained to the bug fix, and coupling Phase 4 aggregation logic into a data migration is fragile.

### Seed paths (D-07, locked from BRIEF)

- **D-07** — Both seed paths MUST tag newly seeded restaurants with the owning `location_id`:
  - `lunchbot/blueprints/setup.py` — install form creates a workspace + default `workspace_locations` row; seed restaurants get that `location_id`.
  - `lunchbot/blueprints/slack_actions.py` — add-office modal submission creates a new `workspace_locations` row; any seed restaurants associated with the new office get that `location_id`.

### Query rewrite (D-08, locked from BRIEF)

- **D-08** — `lunchbot/client/db_client.py::get_candidate_pool` changes signature to `get_candidate_pool(poll_date, channel_id)` (no longer takes workspace_id directly). Implementation:
  1. Resolve `channel_id → location_id` via `channel_locations`
  2. Filter `restaurants` on `r.location_id = :location_id`
  3. Join `restaurant_stats` on `(channel_id = :channel_id AND restaurant_id = r.id)`
  4. Fall back to Beta(1, 1) prior when the `(channel_id, restaurant_id)` row is missing (new channels or post-migration state)

### Reputation job update (D-09, locked from BRIEF)

- **D-09** — The Phase 4 reputation/stats-processing job (scheduled poll-finalize aggregator) must emit `(channel_id, restaurant_id)` rows, not `(workspace_id, restaurant_id)`. Every call site that writes `restaurant_stats` must be updated.

### Testing (D-10, locked from BRIEF)

- **D-10** — Regression tests MUST cover both bugs explicitly:
  - **G-01 isolation test** — two offices under one workspace, two channels each bound to a different office; assert candidate pools do not overlap.
  - **G-02 divergence test** — two channels bound to the **same** office; simulate divergent voting; assert their `restaurant_stats` rows drift independently and Thompson samples reflect the divergence.

### Channel-scheduling restructure (answered inline)

- **D-11 — Polls are channel-bound at insert time.** `polls.slack_channel_id` becomes `NOT NULL`. `upsert_suggestion` signature changes to `upsert_suggestion(poll_date, restaurant_id, workspace_id, slack_channel_id, ...)` — every caller MUST pass the channel. Ad-hoc `/lunch` already knows the channel from slash-command form data; the reputation auto-fill caller reads it from the originating poll row; the scheduled path gets it from the new `channel_schedules` table.

- **D-12 — New `channel_schedules` table.** Columns (minimum): `team_id VARCHAR(64) NOT NULL`, `channel_id VARCHAR(64) NOT NULL`, `schedule_time TIME NOT NULL`, `schedule_timezone VARCHAR(64) NOT NULL`, `schedule_weekdays VARCHAR(32) NOT NULL` (comma-separated like existing `workspaces.poll_schedule_weekdays`), `created_at TIMESTAMPTZ DEFAULT NOW()`. Primary key `(team_id, channel_id)`. `team_id` included for RLS mirror (same pattern as `channel_locations` from migration 007). One schedule per channel; a channel with no row has no schedule.

- **D-13 — Scheduler is channel-keyed.** `scheduler_service.load_all_schedules` reads from `channel_schedules` instead of `workspaces.poll_schedule_*`. Job names become `poll_{team_id}_{channel_id}`. `scheduler_service._run_poll` takes `(team_id, channel_id)` and passes the channel straight through to `push_poll` — no more `poll_channel_for(team_id)` resolution. `update_schedule_job` becomes `update_schedule_job(team_id, channel_id, time_val, timezone, weekdays)` and creates/updates/removes exactly one job. Deleting a channel_schedules row removes its APScheduler job.

- **D-14 — App Home schedule section becomes per-channel.** The single "Poll Schedule" section is replaced by a list of rows, one per `channel_schedules` entry, each showing `#channel — HH:MM TZ — weekdays` with an Edit button. Add a "Schedule a channel" button that opens a modal: channel picker (from a server-side list of channels the bot is a member of — use `conversations.list` with `types=public_channel,private_channel` filtered to `is_member=true`), time input, timezone select, weekday multi-select. Submitting inserts/updates a `channel_schedules` row and calls `update_schedule_job`. The existing "Poll Channel" section is removed (it was a workspace-level single-channel setting — now superseded by per-channel schedules). The existing workspace-level "Poll Schedule" section is removed.

- **D-15 — Migrate existing workspace-level schedules.** In migration 009, for every `workspaces` row where `poll_schedule_time IS NOT NULL`, insert a `channel_schedules` row keyed on `(team_id, workspaces.poll_channel)` — provided `poll_channel` is non-null and resolvable. If `poll_channel` is NULL, skip the row (that workspace had a broken schedule anyway). Preserve `schedule_time`, `schedule_timezone`, `schedule_weekdays` verbatim. After the copy, `DROP` columns `workspaces.poll_schedule_time`, `poll_schedule_timezone`, `poll_schedule_weekdays`, AND `workspaces.poll_channel` (which is now superseded by `channel_schedules`).

- **D-16 — Bot-membership constraint for channel picker.** The App Home "Schedule a channel" modal only lists channels where the bot is currently a member. If a user wants to schedule a channel the bot isn't in, the error path is "add the bot to that channel first." This matches how `/lunch` already works (slash command only invocable where the bot is present).

- **D-17 — Add-office modal must seed restaurants.** Extract the install-flow seeding logic from `setup.py` into a shared helper (e.g., `_seed_office_restaurants(team_id, location_id, lat_lng)`) and call it from both: (a) `setup.py` install form, and (b) `slack_actions.py` add-office modal submission. Every newly created `workspace_locations` row ends up with a seeded candidate pool whose restaurants are tagged with the new `location_id`. Fixes a latent empty-pool bug that would otherwise mask G-01's fix.

### Claude's Discretion

- Exact Alembic migration sequencing (single file vs. multiple revisions) — migration 009 is a single revision unless the planner finds a reason to split.
- Index selection on the new `(channel_id, restaurant_id)` primary key / indices on `restaurants.location_id` — planner picks based on actual query shape.
- Whether to drop any now-unused columns on `restaurant_stats` (e.g. `workspace_id` if present) — planner decides after reading current schema.

</decisions>

<specifics>
## Specific Ideas

- Prior Beta(1, 1) fallback in `get_candidate_pool` means new channels still get a fair sampling on day 1 even though the `restaurant_stats` row does not yet exist.
- Migration 009 naming: follow 008's convention; likely `008_drop_workspaces_location.py` → `009_office_scoped_candidates_and_per_channel_stats.py`.
- `restaurants` backfill for D-03b: join via `workspace_id` → `workspace_locations.is_default = true` → use that `location_id`. This is deterministic because 07.1 guaranteed every workspace has exactly one default office post-migration 008.

</specifics>

<deferred>
## Deferred Ideas

- Retroactive replay of `polls` history into per-channel stats (D-06 rejected for this phase; could be a follow-up if priors take too long to converge in production).
- Clone/split backfill strategies for `restaurant_stats` (D-04 rejected; revisit only if the wipe proves problematic).
- Admin UX for moving a restaurant between offices (out of scope; handled implicitly by delete-and-reseed).
- Cross-office restaurant sharing (explicitly not supported — each restaurant belongs to exactly one office).

</deferred>
