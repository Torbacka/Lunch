# Phase 07.1 Plan 01: Places Autocomplete Client + Proxy Summary

Google Places Autocomplete + Details client functions and public `/places` proxy blueprint with session-token billing and no API-key leakage.

## What Shipped

**Task 1 — Places client functions** (`lunchbot/client/places_client.py`)
- `autocomplete(query, session_token, types='establishment')` — hits Google Places Autocomplete with session token, returns full JSON dict.
- `get_place_details(place_id, session_token=None)` — fetches Place Details restricted to Basic-tier field mask (`place_id,name,formatted_address,geometry/location`).
- `new_session_token()` — returns `uuid.uuid4().hex` for session-based Places billing.
- API key is read exclusively from `current_app.config['GOOGLE_PLACES_API_KEY']`; existing `_check_status` logging pattern reused, key never logged.

**Task 2 — Proxy blueprint** (`lunchbot/blueprints/places_proxy.py`)
- `GET /places/autocomplete?q=...&session_token=...` — returns normalized predictions (`place_id`, `description`, `main_text`, `secondary_text`) plus the session token (generated server-side when absent).
- `GET /places/details?place_id=...&session_token=...` — returns whitelisted fields only (`place_id`, `name`, `formatted_address`, `lat`, `lng`). 404 when geometry missing, 400 when `place_id` missing.
- Registered in `lunchbot/__init__.py` alongside existing blueprints.
- Added `/places/autocomplete` and `/places/details` to `SKIP_PATHS` in the Slack signature middleware so the public install-form proxy is not blocked by signature verification (deviation — see below).

## Threat Mitigations Applied

| Threat ID | Mitigation implemented |
|---|---|
| T-07.1-01 (API key disclosure) | Key only flows through `places_client`; `grep GOOGLE_PLACES_API_KEY lunchbot/blueprints/places_proxy.py` returns 0 matches. Explicit test `test_proxy_does_not_leak_api_key` asserts the key never appears in response bytes. |
| T-07.1-02 (SSRF / tampering) | User input only flows into `input`/`placeid` query params of hardcoded Google URLs. No URL pass-through. |
| T-07.1-04 (over-disclosure) | `/places/details` whitelists 5 fields; raw Google response is discarded. |

## Tests Added

9 tests in `tests/test_places_autocomplete.py`:
- 3 client tests — autocomplete params, details field mask, session token uniqueness.
- 6 proxy tests — predictions response shape, missing `q` (400), details lat/lng whitelist, 404 on missing geometry, missing `place_id` (400), API key non-leak.

All tests pass. No regressions in `tests/test_places.py` or `tests/test_oauth.py`.

- **Found during:** Task 2

**2. [Rule 2 - Missing critical functionality] Signature middleware would have blocked the proxy**
- **Found during:** Task 2
- **Issue:** `lunchbot/middleware/signature.py` runs `verify_slack_signature` as a `before_request` hook on every path not in `SKIP_PATHS`. The plan treats `/places/*` as a public, unauthenticated install-form proxy, but the middleware would have aborted every request with 403 ("Invalid Slack signature") since the browser never produces Slack signatures.
- **Fix:** Added `/places/autocomplete` and `/places/details` to `SKIP_PATHS`, matching the treatment of `/slack/setup`, `/privacy`, `/support`, etc.
- **Files modified:** `lunchbot/middleware/signature.py`
- **Commit:** c206d56
- **Rationale:** Matches threat model disposition T-07.1-05 (Spoofing — accept: no auth needed, only public Google Places data returned).
