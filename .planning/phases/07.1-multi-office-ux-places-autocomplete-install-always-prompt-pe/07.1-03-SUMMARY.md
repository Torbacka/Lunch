# Phase 07.1 Plan 03: Drop legacy workspaces.location column Summary

One-liner: Added Alembic migration 008 with an idempotent `NOT EXISTS`-guarded backfill that promotes any surviving `workspaces.location` value into a `Main Office` row in `workspace_locations` and then hard-drops the `workspaces.location` column entirely.

## What Was Built

`migrations/versions/008_drop_workspace_location_column.py` (revision `008`, down_revision `007`):

- `upgrade()` first inserts a `Main Office` row into `workspace_locations` for every workspace that still has a non-empty `location` AND no existing `workspace_locations` row (idempotent via a `NOT EXISTS` guard so the migration can be safely re-run on a half-applied DB — mitigates T-07.1-11). Then issues `ALTER TABLE workspaces DROP COLUMN location` as a metadata-only operation (T-07.1-12).
- `downgrade()` re-adds `location` as a nullable `VARCHAR(64)` and best-effort populates it from each workspace's current `is_default=TRUE` row in `workspace_locations`. The docstring documents that downgrade is emergency-only per D-21 and that the restored value may not match the original if admins edited offices after the upgrade.

`tests/test_migration_008_drop_location.py` (3 tests) exercises the migration via the existing alembic subprocess pattern used in `tests/test_channel_location_binding.py`:

1. `test_migration_008_drops_location_column` — upgrades to head and asserts `workspaces.location` is absent from `information_schema.columns`.
2. `test_migration_008_downgrade_readds_column` — upgrades, downgrades to 007, asserts the column exists again, then re-upgrades to leave the DB at head for other tests.
3. `test_migration_008_backfills_legacy_location` — downgrades to 007, inserts a legacy `T_LEGACY` workspace with a non-null `location` and zero `workspace_locations` rows, re-upgrades, then asserts exactly one `Main Office` row was created with the correct `lat_lng` and `is_default=TRUE`. Uses `SET app.current_tenant` to pass the RLS filter introduced in migration 007.

## Decisions Made

- **D-19 (hard cutover):** Went straight to `DROP COLUMN` in the same migration as the backfill. No feature flag, no two-step deprecation. Rationale: single-tenant deployment, no multi-version rollout, and plan 02 already eliminated every code read of the column, so a deprecation window adds risk without benefit.
- **D-21 (no rollback window):** Downgrade exists but is documented as emergency-only and lossy. Admins may have renamed/edited offices between 008 and a hypothetical downgrade, and the downgrade does not preserve the prior value verbatim — it re-derives it from the current default office. Acceptable because production rollback is not a supported flow for this phase.
- **Idempotent backfill:** Used `NOT EXISTS (SELECT 1 FROM workspace_locations WHERE team_id = w.team_id)` rather than `ON CONFLICT`. Guarantees the migration is safe to re-run on a half-migrated DB (mitigation for T-07.1-11) and avoids the need for a `UNIQUE` constraint on `(team_id, is_default)` which does not exist.
