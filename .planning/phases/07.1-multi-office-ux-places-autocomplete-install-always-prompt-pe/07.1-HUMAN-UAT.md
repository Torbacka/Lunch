## Gaps

### G-01 — Poll candidate pool pulls from all offices (found during live UAT 2026-04-15)
status: resolved
symptom: With 2+ offices in one workspace, running `/lunch` in a channel bound to office A returns restaurants seeded from office B (and vice versa). Polls are polluted by every office's restaurants.
root_cause: `lunchbot/client/db_client.py` `get_candidate_pool` filters only by `workspace_id` (via tenant RLS). `restaurants` has no `location_id` FK, so there is no way to scope candidates to the channel's bound office.
owner: phase 07.2
fix_direction: Migration 009 adds `restaurants.location_id` FK → `workspace_locations.id`. Backfill tags existing rows to the workspace's default office. Seed paths (`setup.py` install + add-office modal) must set `location_id` on new restaurants. `get_candidate_pool` takes `location_id` + `channel_id` args.

### G-02 — Thompson sampling must be per-channel, not workspace-wide (decided during live UAT 2026-04-15)
status: resolved
symptom: `restaurant_stats` is currently keyed at workspace level, so two channels bound to the same office share alpha/beta history. One team's taste leaks into another team's poll rankings.
decision: alpha/beta/times_shown are keyed by `(channel_id, restaurant_id)`. Candidate pool uses the *channel's* stats, not the workspace's or office's.
owner: phase 07.2
fix_direction: Migration 009 (same migration as G-01) re-keys `restaurant_stats` on `channel_id`. Strategy for existing rows: either drop & re-accumulate, or backfill by splitting the current single-row stats across each channel proportionally (TBD in planning). Phase 4's stats-processing job must be updated to emit per-channel rows.
