## Context at start
- Phase 07.1 was inserted into the roadmap with only a title and `[Urgent work - to be planned]` goal.
- Multi-office schema (`workspace_locations`, `channel_locations`) and the `/lunch` channel-binding prompt had already landed via recent `quick/260415` commits, but `/slack/setup` still wrote the legacy `workspaces.location` column, App Home still exposed the old single-location modal, and there was no admin CRUD UX.

## Round 1 — Scope & UX shape

**Q1. Does my read of Phase 07.1's scope match your intent?**
→ **"Yes, that's the phase"** — full scope including legacy retirement.

**Q2. Where should the admin multi-office UX live?**
→ **"App Home tab (Slack-native)"** — no new web admin surface.

**Q3. Which admin operations belong in this phase (pick the MVP set)?**
→ **"Add + rename + set default (MVP)"** AND **"Include delete office"**.
→ User free-text note: *"When a non admin type /lunch in a new slack channel he/she should be prompted with the questsions of using one of the exiting offices or adding a new"* — adds non-admin office creation from /lunch prompt.

**Q4. How should 'always-prompt per-channel office binding' behave when a workspace has only one office?**
→ User free-text note: *"If its a new channel it should prompt the single office that is configured but ask if you want to add a new office as well."* — overrides all three offered options. The current `resolve_location_for_channel` single-office auto-bind fallback is removed.

## Round 2 — Follow-ups driven by non-admin add-office flow

**Q5. How should Places autocomplete work when a user adds a new office from inside Slack (modal, not web)?**
→ **"Slack external_select backed by Places API"** — in-Slack flow, no browser hop.

**Q6. Who is allowed to add a new office?**
→ User free-text note: *"When the first time a /lunch is invoted into a channel a none admin user is asked if they want an existing office or add a new office. but other time when /lunch is invoked it should remember that its already configured this channel and not prompt."* — reinforces Q4. Any user can add; the binding is remembered after the first /lunch.

**Q7. Delete policy when an office has channels bound to it?**
→ **"Cascade unbind — channels re-prompt on next /lunch"**.

**Q8. What naming should the FIRST office get during OAuth install?**
→ **"Auto-name from Places result"** (e.g. "Spotify HQ, Regeringsgatan 19"). Admin can rename later.

## Round 3 — Legacy retirement & App Home shape

**Q9. How should the legacy workspaces.location column be retired?**
→ **"Hard cutover + drop column now"** — single Alembic migration: backfill workspace_locations from legacy, then drop the column.

**Q10. What shape should the App Home Offices UX take?**
→ **"Replace existing Location section with Offices list"** — each office rendered inline with Rename / Set default / Delete buttons and an Add office button.

## Corrections made to my initial analysis

1. I initially framed "Always prompt even with 1 office" as optional. User's free-text made it mandatory — removed the auto-bind fallback entirely (D-08).
2. I initially planned an admin gate on office creation. User's free-text made this a non-admin flow from /lunch, so D-07/D-11 explicitly carve out office creation from the admin gate.
3. I offered a "keep column as rollback safety net" option. User picked the harder cutover — D-19/D-21 drop the column in the same migration.

## Scope creep redirected
- No scope creep surfaced. Channel-binding management UI (list/move channels per office) was offered, user opted not to include it; captured in Deferred Ideas.
