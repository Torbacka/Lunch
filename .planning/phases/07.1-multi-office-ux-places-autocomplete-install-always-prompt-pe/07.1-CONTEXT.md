# Phase 07.1: multi-office-ux — Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Make multi-office a first-class, end-to-end feature in LunchBot. Specifically:

1. **Install flow** — `/slack/setup` uses Google Places Autocomplete instead of raw "lat, lng" text entry, and creates a `workspace_locations` row on submit (no longer writing the legacy `workspaces.location` column).
2. **Always-prompt per-channel binding** — The first time `/lunch` is invoked in a channel with no binding, any user (admin or not) is prompted to pick an existing office OR add a new one. Subsequent `/lunch` calls in that same channel use the stored binding silently.
3. **Admin Offices UX** — App Home exposes an Offices list replacing the current single-Location section, with Add / Rename / Set default / Delete operations.
4. **Legacy retirement** — The `workspaces.location` column is fully removed from code and schema in this phase.

Out of scope: Phase 7 web-presence pages, marketplace listing copy, billing/freemium gating, anything touching Google Places restaurant search itself (only the office location picker is Places-Autocomplete-ified here).

</domain>

<decisions>
## Implementation Decisions

### Install flow (web, `/slack/setup`)
- **D-01:** Replace the "lat, lng" text input with a Google Places Autocomplete widget (client-side JS Places library or server-side calls to `places_client.py` — planner/research to pick). The user searches by name/address and picks a suggestion; submit sends `place_id` + formatted name + lat/lng.
- **D-02:** On submit, create a `workspace_locations` row with `is_default=true` and auto-name it from the Places result's formatted name (e.g. `"Spotify HQ, Regeringsgatan 19"`). Admin can rename later from App Home.
- **D-03:** The install flow no longer writes `workspaces.location`. That column is dropped in this phase (see "Legacy retirement" below).
- **D-04:** Existing background restaurant-seeding (`_seed_restaurants`) remains wired and is triggered after the workspace_location row is created. It must seed against the newly created office (not the legacy column).

### Always-prompt per-channel binding
- **D-05:** First `/lunch` invocation in a channel with no `channel_locations` row → always prompt the user, regardless of whether the workspace has 1 office or many.
- **D-06:** The prompt offers: (a) existing offices as a static/radio select, AND (b) an "Add a new office" action that opens an in-Slack modal flow.
- **D-07:** Non-admin users are allowed to trigger the "Add a new office" flow from the prompt. There is no admin gate on office creation.
- **D-08:** Once a channel_locations binding exists, subsequent `/lunch` invocations in that channel use the binding silently — no prompt. This means the existing `resolve_location_for_channel` single-office auto-bind fallback is **removed** (contract changes: the resolver no longer auto-binds; it returns None whenever there's no explicit binding).
- **D-09:** If the workspace has zero offices (edge case: admin deleted them all), the prompt shows only "Add a new office" and the /lunch flow is blocked until one is created.

### Add-office-from-Slack flow (non-admin-accessible)
- **D-10:** The "Add a new office" action opens a Slack modal containing:
  - A Slack `external_select` element wired to a new backend endpoint that proxies Google Places Autocomplete queries (similar dispatch pattern to `places_client.py`'s existing usage).
  - On submit: create a new `workspace_locations` row (not default), then immediately bind the current channel to it via `channel_locations`.
- **D-11:** Any Slack user can submit this modal — no permission check. Accidental/duplicate offices are accepted as a tradeoff for frictionless UX (admin can rename/delete later in App Home).

### Admin Offices UX (App Home)
- **D-12:** The current single "Location" section in App Home is replaced by an "Offices" section that lists every `workspace_locations` row for the workspace.
- **D-13:** Each office row shows: name, address (from lat_lng or stored formatted address), a default marker, and action buttons: Rename, Set default, Delete.
- **D-14:** An "Add office" button at the top of the section opens a modal with the same Places-backed `external_select` widget used by the /lunch add-office flow (shared implementation).
- **D-15:** Admin-only operations (Rename, Delete, Set default) are gated on Slack workspace admin status. Add office is NOT gated (any user can add, consistent with D-07/D-11).
- **D-16:** **Delete policy:** Cascade unbind. Deleting an office deletes its `workspace_locations` row and any `channel_locations` rows referencing it. Affected channels will hit the always-prompt on their next `/lunch`.
- **D-17:** **Set default:** Exactly one office is marked `is_default=true` at a time. Setting a new default clears the previous default in the same transaction. The default is used as an initial selection in the always-prompt UI only; it does NOT auto-bind channels.
- **D-18:** Rename edits only the `name` column; address/lat_lng is not editable in this phase (delete + re-add if wrong).

### Legacy retirement
- **D-19:** Write an Alembic migration that: (a) backfills `workspace_locations` from any `workspaces.location` value that does not yet have a corresponding row (as the workspace's first, default office, auto-named "Main Office" or similar), (b) drops the `workspaces.location` column.
- **D-20:** Remove all code reads and writes of `workspaces.location`, including in `workspace_client.get_workspace_settings`, `workspace_client.update_workspace_location`, `workspace_client.update_workspace_settings`'s ALLOWED set, and `app_home_service.build_location_modal` / related Home blocks.
- **D-21:** No rollback safety net window — hard cutover in a single migration. Justification: the new schema has existed since migration 007 and recent code already prefers `resolve_location_for_channel`; legacy reads are dead-ish.

### Cross-cutting
- **D-22:** Google Places API key handling: reuse the existing server-side key from `places_client.py` / env. The web install form's client-side autocomplete, if chosen, must use a key scoped/referrer-restricted; planner/research to confirm the safest approach for a self-hosted single-domain deployment.
- **D-23:** Tenant isolation: every new query path (admin ops, always-prompt, add-office modal) must run under `app.current_tenant` via the existing `_set_tenant` pattern in `workspace_client.py`. No bypass.
- **D-24:** Logging: every admin mutation (create/rename/delete/set-default) emits a structlog event `office_<op>` with `team_id`, `location_id`, `actor_user_id`. Non-admin office creation logs the same shape with `via=lunch_prompt`.
- **D-25:** Tests: extend `tests/test_channel_location_binding.py` and add new test modules for (a) Places-backed external_select handler, (b) App Home Offices rendering and admin gating, (c) cascade-delete behavior, (d) the 007.1 backfill migration.

### Claude's Discretion
- Exact HTML/CSS for the install form's Places autocomplete widget (keep consistent with existing `/slack/setup` styling in `lunchbot/blueprints/setup.py`).
- Whether the install form uses Google's JS Places Autocomplete library directly or a server-side proxy through `places_client.py`. Planner/research should weigh API-key exposure, cost, and accessibility.
- Block Kit layout details for the Offices section in App Home and the Add-office modal — follow patterns already established in `app_home_service.py`.
- Session-token usage and cost optimization for Places Autocomplete calls.
- Error-state copy on the install form and the Slack modal.
- Whether to surface "Address" (formatted) as well as office name in the Offices list — recommended, planner decides exact format.
- Whether the always-prompt uses a radio group, static_select, or button list — pick whatever matches the existing poll-prompt block style in `lunchbot/blueprints/polls.py`.

### Folded Todos
None — no pending todos surfaced for this phase.

</decisions>

<specifics>
## Specific Ideas

- User note (verbatim): *"When a non admin type /lunch in a new slack channel he/she should be prompted with the questsions of using one of the exiting offices or adding a new"* — drives D-05 through D-11.
- User note (verbatim): *"If its a new channel it should prompt the single office that is configured but ask if you want to add a new office as well."* — reinforces that even single-office workspaces show the prompt, overriding the current auto-bind fallback.
- The install form's Places widget should feel like "a real address picker, not a coordinate form" — mirror the affordance of Google Maps search, not the current right-click-to-copy hack.
- Office auto-naming from Places results should preserve the primary name + locality (e.g. "Spotify HQ, Regeringsgatan 19") so admins don't usually need to rename.

</specifics>

- Editing an office's address/lat_lng in place — out of scope; delete + re-add for now (D-18).
- Permission model beyond admin-vs-not (e.g. per-channel owners, least-privilege for rename) — not in scope.
- Places API cost optimization beyond session tokens — planner may choose, but formal budgeting is deferred.
- Multi-office support in the poll-scheduler (if different channels on different offices need separate schedules) — verify not already handled by Phase 5; otherwise, defer.

</deferred>
