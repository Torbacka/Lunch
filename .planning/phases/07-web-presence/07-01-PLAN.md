## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet -> Flask routes | Unauthenticated users access /, /privacy, /support |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing | /slack/install link | accept | The "Add to Slack" button links to the existing OAuth flow which already validates via Slack's servers. No spoofing risk on the link itself. |
| T-07-02 | Information Disclosure | Privacy page | accept | The privacy page intentionally discloses data practices. No secrets are exposed. Table/column names are not mentioned — only user-facing descriptions of data types. |
| T-07-03 | Denial of Service | Static pages | accept | Pages are static HTML strings with no database queries, no computation, no file I/O. Flask's built-in request handling is sufficient. The existing Docker HEALTHCHECK and restart policy (Phase 6) handle container-level issues. |
| T-07-04 | Tampering | SKIP_PATHS bypass | mitigate | Only GET routes for static content are added to SKIP_PATHS. No POST/PUT/DELETE endpoints. No user input is processed. The routes return hardcoded HTML — there is nothing to tamper with. Mitigation: routes only accept GET method (Flask default for @bp.route). |
| T-07-05 | Elevation of Privilege | Public route access | accept | Routes serve only static marketing/legal content. No authentication state, no session data, no workspace context. The tenant middleware runs harmlessly with workspace_id=None. |
</threat_model>
